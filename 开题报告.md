## 第一章 研究意义

### 1.1 背景与问题

三维点云在自动驾驶、机器人、数字孪生等场景中承担“空间几何理解”的关键任务。随着深度学习成为主流建模手段，模型在真实部署中不仅需要高精度，也需要在恶意输入或异常噪声下保持稳定。

对抗机器学习（Adversarial ML）关注的一类核心风险是：攻击者在输入上施加小幅、结构化的扰动，使模型产生明显错误输出。相比 2D 图像，点云数据具有稀疏、离散、几何结构强等特点，导致攻击与防御在工程实现、评价指标与可视化解释上更具挑战。

### 1.2 本课题的意义

本课题拟构建“基于 ART 与 MMDetection3D 的 3D 点云对抗鲁棒性研究平台”，其意义主要体现在：

1. **工程可复现**：将“模型（MMDetection3D）—攻击/防御（ART）—评估与可视化”串成可复现流水线，降低复现实验的门槛。
2. **对比更公平**：用统一的配置、输出格式与记录方式，便于在同一条件下对比不同攻击与防御组合。
3. **结果更可解释**：提供点云样本级的可视化对比，帮助解释“为何失效、哪里失效”。
4. **面向教学与小组作业**：在可控工作量下交付完整链路，形成可展示、可运行、可扩展的成果。

### 1.3 研究目标

本课题的总体目标是：在既有开源工具 ART 与 MMDetection3D 的基础上，搭建一个面向**多模态 3D 感知（LiDAR 点云 + 相机图像）**的对抗鲁棒性研究平台，使研究者能够以统一方式完成“基线—攻击—防御—评估—可视化—复现”的闭环。

为保证目标可执行、可验收，本课题将目标拆解为以下可交付项：

1. **统一实验入口**：用脚本与配置文件驱动实验，避免为不同攻击/防御反复改动业务代码。
2. **多模态样本与报告标准化**：对每次实验输出固定结构的产物（点云/图像样本文件、指标文件、可视化文件、日志）。
3. **基线能力验证（多模态为主）**：在至少一种多模态 3D 检测任务上跑通真实数据加载与模型推理；同时保留单模态（仅点云）作为对照。
4. **攻防对比实验（多模态协议）**：实现“至少 1 种攻击 + 至少 1 种防御 + 至少 1 个多模态模型/数据集”的可复现对比，并覆盖三种攻击设置：点云单攻、图像单攻、联合攻击。

### 1.4 研究问题（Research Questions）

围绕“平台能回答什么问题”，本课题以多模态为主线，拟重点回答以下问题：

1. **融合是否更鲁棒**：在相同扰动预算下，多模态融合模型相较单模态（仅点云对照）是否更鲁棒？提升幅度如何量化？
2. **模态脆弱性对比**：点云模态与图像模态哪个更容易被攻击？当只攻击其中一个模态时，融合模型整体性能下降如何？
3. **联合攻击与级联效应**：当同时攻击点云与图像时，性能是否呈现叠加或放大效应？
4. **防御有效性与代价**：防御能恢复多少性能？其额外代价（训练时长、推理耗时、实现复杂度）是否可接受？
5. **可复现与可扩展**：当更换数据集、融合策略或模型结构时，平台是否能以最小改动继续工作？

### 1.5 研究范围与边界

为控制工作量并保证按时交付，本课题在“多模态为主线”的前提下，明确边界如下：

- **优先聚焦数字域攻击**：即对离线输入施加扰动；物理世界可实现性仅作为讨论与展望。
- **主线任务锁定为多模态 3D 检测**：以 LiDAR + Camera 融合模型为主要对象；单模态（仅点云）作为对照组，而非主线。
- **优先做“平台与评估”**：不以提出新攻击算法为主要目标，而以复现、对比、记录、可视化为核心。
- **控制组合规模**：先完成“点云单攻/图像单攻/联合攻击 × 至少 1 种防御”的最小矩阵，扩展作为加分项。
- **不承诺跨版本长期兼容**：在固定的依赖版本组合上实现稳定运行，并记录版本矩阵。

### 1.6 预期成果形态

本课题最终成果将以“可运行代码 + 可复现实验 + 可展示材料”的形式交付，具体包括：

1. **代码与配置**：统一脚本入口、配置模板、可扩展的适配器接口。
2. **实验产物**：按实验目录组织的样本文件、指标 JSON、可视化图片。
3. **报告与展示**：开题/中期/结题报告、关键对比图、可复现命令清单。
4. **复现指南**：固定依赖版本与运行步骤，确保换机器后仍可重跑。

## 第二章 相关现状调研

### 2.1 研究对象与基本定义

本课题以**多模态 3D 目标检测**作为主要研究对象，并保留单模态任务作为对照：

- **多模态 3D 目标检测（主线）**：输入 LiDAR 点云 + 相机图像，输出 3D 边界框与类别。
- **点云分类或单模态检测（对照）**：输入点云，输出类别或 3D 框，用于对比“融合是否更鲁棒”。

“攻击—防御—评估”在本课题中采用如下定义：

- **干净样本（clean）**：原始点云或仅含自然噪声。
- **对抗样本（adversarial）**：在不改变语义目标的前提下，通过优化扰动使模型输出显著劣化的输入。
- **防御样本（defended）**：对输入做预处理或对模型做鲁棒化训练后得到的结果，用于抵消攻击影响。

多模态场景下还需补充：

- **单模态攻击**：只攻击点云或只攻击图像，另一模态保持干净。
- **联合攻击**：同时对点云与图像施加扰动，在各自预算约束内追求整体失效。

为避免概念混淆，本课题还将补充两点约定：

1. **攻击威胁模型（Threat Model）**：默认以白盒攻击为主（即攻击者可获得模型结构与梯度），在条件允许时再扩展到黑盒设置（查询/迁移攻击）。
2. **扰动预算与约束**：攻击必须在约定预算内（例如坐标扰动的 $L_\infty$ 或 $L_2$ 约束），并尽量保持几何合理性（例如避免产生大量离群点）。

### 2.2 攻击方法概览

点云对抗攻击通常以“约束扰动大小、保持几何合理性”为目标，常见思路包括：

1. **梯度型扰动**：在点坐标或特征上沿梯度方向迭代更新，追求攻击成功率。
2. **结构型扰动**：通过点的删除、插入、局部扭曲等方式在几何层面破坏模型感知。
3. **任务相关攻击目标**：分类强调类别翻转；检测强调漏检、误检、定位偏移等。

从平台工程角度，攻击实现需要同时考虑三个层面：

- **输入空间**：点坐标（XYZ）、强度/反射率（如有）、附加特征（如法向量、颜色等）。
- **目标函数**：分类可用交叉熵；检测可能涉及分类分支 + 回归分支 + NMS 后处理的综合效果。
- **可比性约束**：在相同预算、相同随机种子、相同评估协议下输出结果，避免“换了攻击也换了评估”的不可比问题。

### 2.3 防御方法概览

防御通常分为两条路线：

1. **输入端防御**：去噪、重采样、离群点剔除、平滑等，在不改模型的情况下提升鲁棒性。
2. **模型端防御**：对抗训练、鲁棒正则、蒸馏与集成等，通过训练策略提升对扰动的抵抗能力。

在小组作业的可控范围内，本课题优先实现两类“容易验收”的防御：

- **输入端轻量防御**：例如简单去噪/裁剪/重采样等，可直接作用于 `.npy` 点云。
- **训练端基线防御**：例如对抗训练的最小实现版本（在少量样本/少量 epoch 下跑通流程并记录代价）。

### 2.4 工具与平台现状

现有生态中，ART 提供相对统一的攻击/防御 API，而 MMDetection3D 提供成熟的 3D 任务训练推理框架。二者组合的难点主要在“接口对接”和“实验规范化”上：如何把框架内的模型与数据流映射到对抗攻击需要的输入形式、损失与梯度回传路径，并且稳定产出可对比的日志、样本与可视化。

因此，本课题的核心价值并非提出全新的攻击算法，而是**做出一套可复现、可对比、可扩展的研究平台**。

### 2.5 评估指标与报告规范（平台需要补齐的“共识”）

如果没有统一指标与报告格式，攻击/防御实验即使跑通也难以形成可比较的结论。因此平台将明确如下指标与记录规范：

1. **分类任务指标**
	- Top-1 Accuracy（干净/攻击/防御三种设置分别统计）
	- Attack Success Rate（ASR）：攻击导致预测错误的比例

2. **检测任务指标（后续扩展）**
	- mAP（attack 前后对比）
	- 漏检率/误检率（以阈值规则或评估脚本输出为准）
	- 定位偏移（例如中心点偏差、尺寸偏差的统计摘要）

3. **代价指标（不可忽视）**
	- 生成对抗样本的时间开销（每样本/每 batch）
	- 防御的训练/推理额外开销
	- 产物体积（样本文件大小、日志大小）

4. **报告规范**
	- 每次实验输出：配置快照、指标 JSON、样本文件路径、运行环境信息（版本/设备）
	- 统一命名与目录结构，便于后续批量汇总与画图

### 2.6 小结

本章给出了概念边界、攻防基本路线与平台化落地时需要的指标与报告规范。后续章节将把这些“抽象要求”映射到具体的工具选型与工程实现。

### 2.7 点云数据与任务的工程特性（为什么平台化更难）

与图像不同，点云的工程特性会直接影响攻击、防御与评估的实现方式：

1. **无序性（Permutation Invariance）**：点的顺序不影响语义，攻击不应依赖“点的排列”。
2. **稀疏与不均匀采样**：点密度随距离变化，局部扰动可能造成显著结构破坏。
3. **坐标尺度与单位**：不同数据集坐标系、单位、原点定义差异大，平台需明确归一化/坐标系约定。
4. **下游后处理复杂**：检测任务常含 NMS、阈值筛选等非光滑步骤，导致端到端梯度与攻击目标更难定义。

因此，“同一种攻击在不同模型/数据集上能否公平比较”很大程度依赖平台的规范化能力。

### 2.8 多模态作为主要方向（导师指定切入点）

多模态指在 3D 感知任务中同时使用多种输入（例如 LiDAR 点云 + 相机图像）。本课题将多模态作为主要方向，原因在于：

1. **贴近真实应用形态**：自动驾驶等场景天然是多传感器系统，多模态鲁棒性更具现实意义。
2. **问题更“可研究”**：融合模型在单一模态受攻击时的“补偿能力”、以及联合攻击下的“级联失效”是可量化、可对比的问题。
3. **更利于平台贡献**：平台能够沉淀多模态的攻击设置、预算约束、可视化与指标规范，形成可复用的实验协议。

多模态也会带来平台侧的额外挑战：

- **攻击协议更复杂**：需要分别定义点云预算与图像预算，并明确联合攻击的目标与约束。
- **融合策略影响鲁棒性**：不同融合位置（早/中/晚）可能导致对攻击的敏感性不同，平台需要将其作为可配置因素写入报告。

因此，本课题将采用“多模态为主线 + 单模态为对照”的策略：既确保研究方向对齐导师，也确保结论可解释、可比较。

## 第三章 所选择工具初步搭建情况

### 3.1 工具选择与分工

- **ART（Adversarial Robustness Toolbox）**：提供攻击与防御的统一接口与基线实现。
- **MMDetection3D**：提供点云分类/检测的工程框架与模型实现。
- **NumPy + 可视化工具链**：用于样本保存、对比与展示。

面向多模态主线，平台在工程上需要同时处理两类输入：

1. **点云分支**：点坐标/特征的扰动、保存与可视化。
2. **图像分支**：图像扰动的生成、保存与可视化（可先以最小实现跑通，再逐步完善）。

平台采用“配置驱动 + 脚本入口”的组织方式，保证实验可复现与可迁移。

### 3.2 平台总体设计（模块与数据流）

平台设计遵循“输入输出可控、过程可记录、结果可复现”的原则，核心数据流如下：

1. **数据准备**：从数据集得到点云与图像输入，并统一成平台内部表示（点云数组 + 图像数组/张量 + 标注）。
2. **基线推理/评估**：在干净输入上跑一次，产出基线指标与可视化。
3. **攻击生成**：支持三种设置：点云单攻、图像单攻、联合攻击；分别在各自预算约束内生成对抗输入，并保存为可复现文件。
4. **防御处理**：可分别对点云与图像做输入端防御，或对融合模型做训练端防御（两者可独立开启）。
5. **统一评估与可视化**：对不同攻击设置下的 clean/adversarial/defended 输出同一套指标、对比图与报告。

为了让平台既适合课堂展示又便于扩展，平台约定一套最小“实验产物清单”：

- `clean.npy`：干净点云样本
- `adversarial.npy`：对抗点云样本
- `defended.npy`：防御后的点云样本
- `report.json`：本次实验的核心指标、参数摘要、路径索引
- `comparison_3d.png` / `comparison_2d_xy.png`：可视化对比图

此外，平台鼓励在 `report.json` 中记录能支撑复现的最小字段集合。建议字段示例如下（字段以实际实现为准）：

```json
{
	"task": "classification",
	"n_samples": 1,
	"attack": {"enabled": true, "name": "fgsm", "epsilon": 0.03, "steps": 1},
	"defense": {"enabled": true, "name": "denoise", "params": {"k": 16}},
	"metrics": {"acc_clean": 1.0, "acc_adv": 0.0, "acc_defended": 1.0, "asr": 1.0},
	"artifacts": {
		"clean": "clean.npy",
		"adversarial": "adversarial.npy",
		"defended": "defended.npy",
		"vis_3d": "comparison_3d.png",
		"vis_2d": "comparison_2d_xy.png"
	},
	"env": {"python": "3.x", "torch": "2.x", "device": "cuda"}
}
```

该字段集合的目的不是追求“面面俱到”，而是确保：

- 外部读者在没有上下文的情况下也能看懂实验做了什么
- 后续批量汇总时可以自动抽取关键指标与参数

### 3.3 当前完成情况（以仓库实际为准）

已在本仓库完成以下可运行闭环：

1. **项目结构与文档体系**：形成 docs、scripts、configs、outputs 等清晰目录，便于协作与复现。
2. **统一流水线脚本**：通过 [scripts/run_pointcloud_pipeline.py](scripts/run_pointcloud_pipeline.py) 生成并保存三类样本（clean/adversarial/defended）与报告文件。
3. **可视化对比工具**：通过 [scripts/visualize_pointcloud_samples.py](scripts/visualize_pointcloud_samples.py) 生成三维对比图与二维投影图，输出到 outputs 目录。
4. **配置模板**：通过 configs 下的模板文件组织实验参数，降低“改代码做实验”的成本。

#### 3.3.1 当前可复现的运行方式

当前仓库已经具备“命令 → 产物”的演示闭环。典型流程为：

1. 运行流水线脚本生成样本（clean/adversarial/defended）与报告文件。
2. 运行可视化脚本生成对比图。

仓库中对应的入口脚本分别是：

- [scripts/run_pointcloud_pipeline.py](scripts/run_pointcloud_pipeline.py)
- [scripts/visualize_pointcloud_samples.py](scripts/visualize_pointcloud_samples.py)

这使得开题阶段就可以用“真实命令与真实产物”支撑汇报，而不是停留在概念层面。

#### 3.3.2 可复现规范（建议写入最终报告/提交材料）

为避免同一实验在不同机器/不同同学环境下出现差异，建议遵循：

1. **固定依赖版本**：使用冻结的依赖文件与固定的 Docker/环境脚本。
2. **固定随机种子**：数据抽样、扰动初始化、训练过程尽量固定种子并记录。
3. **配置快照**：每次运行把配置文件复制到输出目录，确保“运行时参数”可追溯。
4. **输出目录不复用**：每次实验使用新的输出目录，避免产物混淆。

当前产出能够支撑课堂/小组作业层面的演示：

- 一键运行流水线生成结果
- 可视化直观对比“干净/攻击/防御”差异
- 具备后续接入真实数据集与真实模型的接口位置

补充说明：目前平台已经满足“从一个入口脚本产出全套产物”的最低可验收标准，并且各产物之间存在明确的路径与命名关联，便于在汇报中展示“从命令到结果”的完整链路。

### 3.4 关键接口与可扩展点（面向后续工作）

为支持后续接入真实数据集与更多攻防方法，平台预留以下扩展点：

1. **数据适配器（Dataset Adapter）**
	- 输入：原始数据集样本（点云文件、标注、可能的图像）
	- 输出：统一的点云数组与标签/目标结构
	- 要求：固定随机种子后可复现采样结果

2. **攻击适配器（Attack Adapter）**
	- 输入：模型、损失函数、点云样本、攻击预算
	- 输出：对抗点云与攻击元数据（迭代次数、预算、成功与否）
	- 要求：对不同模型尽量保持统一接口

3. **防御适配器（Defense Adapter）**
	- 输入端：点云预处理函数（去噪、裁剪、重采样等）
	- 训练端：训练策略封装（对抗训练的最小闭环）

4. **评估与汇总器（Evaluator / Aggregator）**
	- 输出统一 JSON 格式指标
	- 支持批量汇总生成对比表与图

### 3.5 仍需完成的关键工作

为满足“研究平台”的目标，还需进一步完成：

1. **真实数据集接入**：将 data 目录填充为可用数据集，并验证加载与评估流程。
2. **与 MMDetection3D 模型的更紧密对接**：针对检测任务构建可攻击的端到端输入与目标函数。
3. **评估指标完善**：在分类准确率之外，补充检测任务的漏检/误检/定位偏移等指标，并统一输出。
4. **批量实验与对比表**：形成“攻击×防御”的实验矩阵与总结图表。

### 3.6 阶段性验收标准（建议）

为保证每一步可验收、可展示，本课题建议采用以下验收标准：

1. **平台可运行**：在固定环境中可一键生成 clean/adversarial/defended 产物与可视化。
2. **指标可读**：`report.json` 中包含最少必要字段（任务类型、样本数量、攻击预算、防御开关、主要指标）。
3. **结果可对比**：同一输入样本在三种设置下可被并排展示，并能解释差异来源。
4. **实验可复现**：同一配置与随机种子下能得到一致或高度一致的统计结果（允许浮动但需可解释）。

### 4.2 实验设计与评价方案

为避免“跑通但不可比”，本课题拟采用以下实验设计原则：

1. **控制变量**
	- 固定数据划分、随机种子、模型权重（或训练协议）
	- 攻击预算与迭代次数固定后再做横向对比

2. **最小实验矩阵（多模态主线，可在时间允许时扩展）**
	- 任务：多模态 3D 检测（LiDAR+Camera 融合）
	- 对照：单模态（仅点云）或“只输入点云的同类模型”作为对照基线
	- 攻击：至少 1 种梯度型攻击作为基线
	- 防御：至少 1 种输入端防御作为基线
	- 攻击设置：点云单攻、图像单攻、联合攻击
	- 组合：每种攻击设置都需要 clean、attack、attack+defense 三种状态

3. **输出与汇总**
	- 单次实验：产出样本与 `report.json`
	- 批量实验：额外产出汇总表（CSV/JSON 均可）与汇总图（如柱状图/折线图）

#### 4.2.1 评价“融合是否更鲁棒、以及防御是否值得”的判定框架（建议）

仅比较单一指标往往不够，本课题建议分两层做判定：

1. **融合鲁棒性判定（Fusion Robustness）**
	- 在相同预算下，对比“多模态融合模型”与“单模态对照模型”的性能下降幅度
	- 分别统计：点云单攻、图像单攻、联合攻击三种设置

2. **防御性价比判定（Defense Cost-Effectiveness）**
	- 在攻击设置固定时，比较“有无防御”带来的恢复幅度与代价

- **效果（Effectiveness）**：防御使攻击后指标恢复多少（例如从 30% 恢复到 60%）。
- **代价（Cost）**：
	- 输入端防御：推理前处理耗时是否增加、是否引入明显点云失真
	- 训练端防御：训练时间是否显著增加、是否需要额外数据

最终输出可以是一个简洁表格：每种防御在“恢复幅度/额外开销/实现复杂度”上的打分或排序，用于答辩时清晰表达。

### 4.3 资源需求与可行性

1. **软件环境**：Python + PyTorch + ART + MMDetection3D（已完成基础搭建）。
2. **硬件环境**：建议具备 NVIDIA GPU（用于更快完成模型推理与对抗样本生成）；在资源不足时可用小样本 + 低迭代次数保证演示。
3. **数据资源**：至少准备一个可公开获取的数据集（或先以合成数据验证流程）。

### 4.4 风险分析与应对

| 风险点 | 可能影响 | 应对策略 |
|---|---|---|
| 数据集下载/预处理耗时 | 推迟真实数据接入 | 先用合成/小样本验证，数据下载并行推进 |
| 多模态检测端到端梯度/目标函数实现复杂 | 多模态主线进度受阻 | 先保证多模态基线推理与评估跑通；攻击从单模态（点云或图像）切入，再扩展联合攻击 |
| 运行环境差异导致不可复现 | 结果不一致影响汇报 | 固定依赖版本与运行命令，输出环境信息到报告 |
| 实验矩阵过大导致时间不足 | 难以按期完成 | 先保证“1 攻击 + 1 防御 + 1 任务”最小矩阵，扩展作为加分项 |

### 4.6 预期创新点与贡献（面向小组作业）

本课题的“创新点”以工程平台化贡献为主，强调可复现与可比较：

1. **统一对抗评估入口**：把攻防与评估组织成固定流程，降低入门门槛。
2. **标准化实验产物**：清晰的样本、指标、可视化三件套，便于复核与复现。
3. **可解释的可视化输出**：以样本级对比图支撑结论表达，而非只给出单一指标。
4. **可扩展的模块边界**：用适配器与配置模板支撑后续扩展（更多攻击/防御/数据集）。

### 4.7 预期成效

本课题的预期成效以“平台可运行、对比可复现、结果可解释”为主：

1. **可运行平台**：提供统一入口脚本与配置模板，可在同一环境中复现实验。
2. **样本级可视化**：输出清晰的对比图，支持展示攻击与防御对点云结构的影响。
3. **指标与报告**：输出统一格式的实验报告，支持横向比较。
4. **可扩展性**：后续可在同一框架下加入更多攻击/防御算法与数据集。

进一步的量化成果（以实际实验为准）将包含：

- 不同攻击预算下的性能下降曲线
- 不同防御策略下的恢复幅度与开销对比
- 样本级可视化案例集（用于答辩展示）
